    BB600 OCL

    The BB600.ocl36.txt OCL program is a comprehensive Operation Control Language (OCL) script for an IBM System/36 or AS/400 environment, serving as the main control program for the invoice posting workflow. It orchestrates a series of steps to process invoice batches, manage EDI (Electronic Data Interchange) files, print registers, post sales analysis transactions, and clean up temporary files. Below is a detailed explanation of the process steps, business rules, external programs called, and tables (files) used.

    Process Steps of the BB600 OCL Program¶

    The BB600.ocl36.txt OCL program coordinates multiple tasks in the invoice posting process, invoking several RPG and OCL programs, managing EDI files, and performing file maintenance. The steps are as follows:

    1. Wait for EDISNDCK Process:
    2. The program checks if the EDISNDCK job is active (IF ACTIVE-'EDISNDCK').
    3. If active, it waits for 15 seconds (WAIT INTERVAL-000015) and loops back to the TRYAGN tag to check again.

    4. This ensures that no conflicting EDI send processes are running before proceeding.

    5. Create JEREMY Table for Order Sync Protection:

    6. The // IFF DATAF1-?9?JEREMY BLDFILE ?9?JEREMY,S,RECORDS,999000,10,,T,,,NDFILE statement checks if the JEREMY file exists. If it does not, it creates a temporary file (?9?JEREMY) with a capacity of 999,000 records, 10 bytes per record, and type T (temporary).

    7. This file is used to prevent order synchronization during invoice posting, as noted in the comments.

    8. Invoke BB630 for Sales Journal Update:

    9. The // BB630 ,,,,,,,,?9? statement calls the BB630 OCL program (likely BB630.ocl36.txt), which executes the BB630 RPG program to update the Sales Journal number (ACSLJ#) in the ARCONT file for company number 10.

    10. Set System Attributes and Date Handling:

    11. The // GSY2K statement enables Year 2000-compliant date handling for proper processing of two-digit year formats.

    12. The // ATTR CANCEL-NO,INQUIRY-YES statement sets the job attributes to prevent cancellation (CANCEL-NO) and allow inquiry (INQUIRY-YES).

    13. Process EDI 810 Documents:

    14. Check for EDI 810 Data:
      • The // IFF DATAF1-?9?EDOT?20? GOTO JUMP statement checks if the batch-specific EDI output file (?9?EDOT?20?) exists. If not, it skips to the JUMP tag, bypassing EDI 810 processing.
    15. Run BB513:
      • The // LOAD BB513 block loads and executes the BB513 RPG program, opening:
      • EDIOUT (?9?EDOT?20?, shared access): Temporary EDI work file.
      • EDI810 (?9?EDI810, shared access): Permanent EDI invoice file.
      • BB513 copies records from EDIOUT to EDI810 for EDI 810 invoice transmission.
    16. Run BB514:
      • The // BB514 ,,,,,,,,?9? statement calls the BB514 OCL program, which executes the BB514 RPG program to combine FIL855 (EDI 855 purchase order acknowledgments) and EDI810 (EDI 810 invoices) into EDISEND for transmission to Kleinschmidt.

    17. Run BB517:

      • The // LOAD BB517 block loads and executes the BB517 RPG program, opening:
      • EDIOTH (?9?EDOTH?20?, shared access): Temporary EDI invoice header file.
      • EDIINVH (?9?EDIINVH, shared access): Permanent EDI invoice header file.
      • EDIDPIH (?9?EDIDPIH, shared access): Duplicate EDI invoice header file.
      • BB517 processes EDIOTH records, adding new records to EDIINVH, copying duplicates to EDIDPIH, and logging duplicates to a printer file.

    18. Process EDI 945 Documents:

    19. Check for EDI 945 Data:
      • The // IFF DATAF1-?9?EDWSA?20? GOTO JUMP1 statement checks if the batch-specific EDI warehouse shipping advice file (?9?EDWSA?20?) exists. If not, it skips to the JUMP1 tag, bypassing EDI 945 processing.

    20. Run BB516:

      • The // BB516 ,,,,,,,,?9?,,,,,,,,,,,?20? statement calls the BB516 OCL program, which executes the BB516 RPG program to copy records from EDWSA (?9?EDWSA?20?) to EDI945 (?9?EDI945) for EDI 945 warehouse shipping advice transmission.

    21. Print Registers:

    22. The // SWITCH 1XXXXXXX statement sets switch 1 to suppress the gross profit report.
    23. The // LOAD BB600 block loads and executes the BB600 RPG program (not to be confused with this OCL program), opening:
      • INVTOT (?9?BBTT?20?): Invoice totals file.
      • BBTRAN (?9?BBTR?20?): Invoice transaction file.
      • BICONT (?9?BICONT, shared access): Billing control file.
      • ARCUST (?9?ARCUST, shared access): A/R customer file.
      • Printer files: LIST, LIST2, MILIST (device P5, priority 0), MILISTP1 (device P1, priority 0).
    24. Conditional printer overrides for the REPORT file based on the ?9? environment variable:
      • If ?9? = G, OVRPRTF FILE(REPORT) OUTQ(QUSRSYS/SJPOST).
      • If ?9? = G and another condition, OVRPRTF FILE(REPORT) OUTQ(QUSRSYS/TESTOUTQ).
    25. The BB600 RPG program likely prints invoice registers or reports using these files.

    26. The // SWITCH 0XXXXXXX statement resets switch 1 after execution.

    27. Post Sales Analysis Transactions:

    28. The // LOAD BB605 block loads and executes the BB605 RPG program, opening multiple files:
      • SA5TRN (?9?BBSA?20?): Sales analysis transaction file.
      • SA5TSH (?9?BBSS?20?): Sales analysis transaction header file.
      • SHPADR (?9?SHPADR, shared access): Ship-to address file.
      • SA5FILD, SA5FILM, SA5DBBD, SA5DBBM, SA5BCMD, SA5BCMM, SA5COPD, SA5COPM, SA5SHA (?9? prefix, shared access): Various sales analysis files.
      • BBIBCH, BBIBCHX (?9? prefix, shared access): Invoice batch header files.

    Business Rules¶

    1. Process Synchronization:

    2. The program waits for the EDISNDCK job to complete to avoid conflicts with EDI send processes, ensuring data integrity during invoice posting.

    3. Order Sync Protection:

    4. The JEREMY file is created to prevent order synchronization during invoice posting, ensuring that invoice processing is isolated from order updates.

    5. EDI Processing:

    6. EDI 810 (invoices) and EDI 945 (warehouse shipping advice) files are processed only if their respective batch-specific files (EDOT?20?, EDWSA?20?) exist.
    7. Records are copied to permanent files (EDI810, EDI945, EDIINVH, SA5IVEI) and duplicates are managed (EDIDPIH).

    8. Files are archived (via BB514 and BB516) to maintain historical data for up to 20 versions.

    9. Sales Journal Update:

    10. The Sales Journal number is updated for company 10 via BB630, ensuring unique journal entries for A/R posting.

    11. Reporting:

    12. Invoice registers are printed using BB600, with the gross profit report suppressed (switch 1).

    13. Output is directed to specific printer queues based on the environment (?9? = G).

    14. Sales Analysis:

    15. Sales analysis transactions are posted to multiple files via BB605, ensuring accurate tracking of sales data.

    16. File Cleanup:

    17. Empty transaction files (BBTR?20?, BBTX?20?) are reorganized and deleted if empty.
    18. All batch-specific temporary files are deleted after processing to maintain system efficiency.

    19. The EDWSA file is cleared after EDI 945 processing.

    20. Spoolflex Integration:

    21. For Viscosity Post (?13? = PP), ASN and invoice data are prepared for Spoolflex distribution, ensuring copies are available for splitting and distribution.

    22. Conditional Processing:

    23. Many steps are conditional based on file existence (DATAF1) or environment variables (?9?, ?13?), allowing flexibility for different configurations (e.g., PP for Viscosity Post, G for specific output queues).
